Abstract

DrO (Soft Drink Web Organizer) represents a modern implementation of a beverage exploration and management system, structured using the Scholarly HTML document model. This web-based platform enables users to authenticate, browse curated drink collections, create personalized lists, and interact with drink-related content through a responsive and modular interface. Designed entirely with standard web technologies—HTML5, CSS3, and vanilla JavaScript — DrO emphasizes clean architecture, semantic navigation, and seamless integration with RESTful APIs. This documentation outlines the structure, encoding, and core functionalities of the system, highlighting its adherence to open web standards and its interoperability across modern browsers. By leveraging Scholarly HTML, this website contributes to a more structured and accessible digital ecosystem in the domain of beverage organization and social consumption.

This webpage is written in accordance with the Scholarly HTML template — a domain-specific rich document format. The role of this documentation is to present the essential functionalities of our website and its user interaction. The structure follows the IEEE System Requirements Specification Template.

For detailed information on these templates, please refer to the Scholarly HTML and IEEE System Requirements Specification Template.

1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) document outlines the specifications for the development of a web-based application entitled DrinkApp. The purpose of this document is to clearly define the scope, objectives, and goals of the project, which revolves around the exploration, organization, and personalized management of drinks within a user-friendly digital environment. DrinkApp is designed to allow users to browse drink collections, save favorites, create and manage custom drink lists, and interact with structured drink-related content through a responsive and accessible interface.

This document also serves as a guide for developers and maintainers by detailing the structure of the frontend system, the interaction with backend APIs, and the behavior of each functional module such as authentication, drink filtering, and list operations. Additionally, it adheres to the IEEE System Requirements Specification format and is encoded in Scholarly HTML to ensure consistency, semantic clarity, and ease of integration in academic and professional contexts.

1.2 Document Conventions

This document follows standard Software Requirements Specification conventions, including hierarchical section numbering, descriptive headings, and structured formatting for ease of reading and reference. The layout is aligned with the IEEE SRS template and encoded in Scholarly HTML to ensure semantic clarity and interoperability across modern browsers and academic tools.

The text is displayed using the Roboto font with a sans-serif fallback to maintain visual consistency across platforms. Inline code elements such as API endpoints, file names, or variable names are presented in a monospace font. Where appropriate, semantic HTML5 tags are used to structure the document clearly and meaningfully, enhancing both modularity and compatibility with automated processing tools.

1.3 Intended Audience and Reading Suggestions

This document is intended for individuals who enjoy discovering and organizing drinks, whether they are casual users looking for inspiration or enthusiasts who wish to keep track of their favorite beverages. The application addresses anyone interested in exploring drinks by category or taste, creating personalized drink lists, and saving favorites in a structured and visually appealing environment. Moreover, the website is designed to be accessible and intuitive for users of all ages, from teenagers to older adults.

Readers are encouraged to begin with the Introduction section to understand the purpose and context of the application, followed by the functional and technical sections that match their interests. End users may focus on the usage-related chapters, while those with a more technical background can refer to the implementation and requirements sections for further insight.

1.4 Product Scope

Our goal is to bring together a community of drink enthusiasts in a space that encourages exploration and personal expression. The website is meant to be a casual and enjoyable platform for users who are passionate about beverages, offering them a place to discover new drinks, save their favorites, and create personalized drink lists.

The objective is to give users an easy and organized way to manage their drink preferences, browse curated selections, and share their taste with others. At the same time, the development of this platform allowed us to deepen our understanding of web technologies by applying them in a real-world context aligned with a theme we genuinely care about.

2. Overall Description

2.1 Product Perspective

DrinkApp is a stand-alone web application designed to function independently, without requiring integration with external platforms or third-party systems. It represents a complete, modular product that covers all core functionalities related to user authentication, drink browsing, personal list management, and interaction with a backend through standardized RESTful APIs. Its structure is built around the principle of modularity, where each page operates as an independent functional unit within the broader system.

While there are other applications that catalog drinks or offer cocktail recipes, DrinkApp focuses on the personalized organization of beverages, giving users full control over their collections and preferences. The application is targeted towards casual users and drink enthusiasts rather than commercial or enterprise environments. Its simplicity, responsive design, and intuitive navigation make it suitable for both desktop and mobile usage, without relying on external libraries or frameworks.

Our goals are to:

  • Provide users with a clean, intuitive interface for discovering and organizing drinks.
  • Ensure full compatibility with all major modern web browsers.
  • Allow users to create, manage, and personalize drink lists with ease.
  • Facilitate efficient drink filtering and searching based on tags or categories.
  • Enable seamless interaction with the backend through secure API communication.
  • Offer a lightweight and responsive experience on both desktop and mobile devices.
  • Encourage users to explore new beverages and share their preferences in a structured way.

2.2 Product Functions

DrO Website offers users the ability to create a personal account through a secure registration and login process. Once authenticated, users gain access to personalized features such as managing their own drink collections and saving preferences. The authentication system ensures that each user has a unique profile, which stores their custom lists and favorite drinks.

One of the core features of the application is the drink discovery feed. Users can browse a curated list of drinks, which can be filtered based on predefined categories such as "sodas", "waters", "lemonade","teas" and so on. Also, the drinks can be sorted by a nutrition score such as "A", "B", "C", "D" and/or "E". In addition to this, you can select a quantity interval for these drinks between 0 and 5000 ml. There is also a functionality to sort the drinks by different criterias, like "name","brand","nutrition grade","price". Each drink item includes a panel with detailed information such as name, brand, quantity, nutrition score and price, allowing users to make informed choices about what they want to save or try.

Users can interact with drinks by adding them to a list of favorites, making it easy to revisit them later. Additionally, the application allows the creation and management of custom drink lists. These lists can be named, edited, and populated with selected drinks based on personal taste or thematic groupings.

Drink lists can be modified at any time: users may rename them, delete them, or remove specific drinks from them. Depending on the implementation stage, lists can also be set as public or private, giving users control over how their collections are shared or displayed. The application maintains a clear and responsive interface to support these interactions seamlessly across devices.

The application includes group-related features that allow users to join or create groups where information and personal imperssions about drinks can be shared among members.

In addition to managing drinks and lists, users also have access to a profile section where they can update their personal information. They can change their profile picture, edit their display name, email address, and personal description, as well as update their account password. For full control over their data, the application also provides the option to permanently delete the account, which results in the removal of all associated content such as saved drinks and personal lists.

Users with administrative privileges have access to a dedicated admin panel where they can perform high-level management operations. Administrators can view and delete user accounts, remove entire groups created by users, or delete drinks from the global drink database when necessary. These actions help maintain the integrity of the platform and allow moderation of content that may violate usage guidelines or clutter the system with redundant data.

The application also features a ranking system that displays the most popular drinks based on how frequently they appear in users' favorite lists. This leaderboard offers a real-time overview of trending beverages within the community and encourages users to explore what others are enjoying. The ranking is automatically updated as users add or remove drinks from their favorites, providing a dynamic and relevant reflection of collective preferences.

DrO website also includes a statistics page where users can visualize aggregated data about the available drinks. These statistics are generated based on attributes such as quantity (e.g., volume in milliliters), nutrition grade, and assigned tags. This feature adds an analytical dimension to the application and enhances the exploratory experience.

2.3 User Classes and Characteristics

The primary user class of DrinkApp is composed of regular users who access the platform for personal use. These users can create an account, log in securely, and interact with the main features of the application such as browsing drinks, filtering by tags, saving favorites, and managing custom drink lists.

A secondary user class is represented by administrators, who have access to an extended set of functionalities through a dedicated admin panel. These users are responsible for overseeing the integrity of the system by managing user accounts, moderating content, and removing inappropriate or redundant data such as duplicate drinks or inactive groups. Administrator access is restricted and requires elevated privileges assigned manually.

2.4 Operating Environment

DrinkApp is a web-based application that requires only a modern web browser and an active internet connection to function. It is designed to run on a wide range of client devices, including desktop computers, laptops, tablets, and smartphones.

The application is fully compatible with operating systems such as Windows, macOS, Linux, Android, and iOS, as long as a supported browser is available. Recommended browsers include Google Chrome, Mozilla Firefox, Microsoft Edge, Safari and Opera. No installation is required, and the application runs entirely in the browser environment using client-side technologies.

2.5 Design and Implementation Constraints

The development of DrinkApp was carried out under several deliberate design and implementation constraints. The most significant constraint was the requirement to avoid the use of any frontend frameworks or libraries. The entire application was developed using only native web technologies — HTML5, CSS3, and vanilla JavaScript — to encourage a deep understanding of core browser capabilities and to reduce external dependencies.

Additionally, the frontend operates strictly on the client side and relies on RESTful API calls to communicate with the backend. This separation imposes certain design considerations, such as handling asynchronous requests manually and managing application state without the help of external tools. All frontend pages follow a "Pretty URLs" folder structure, which requires server-side support for routing if deployed on environments that do not natively handle static directory indexing.

2.6 User Documentations

DrinkApp provides two dedicated documentation pages designed to assist users in understanding and navigating the application: the About page and the Help page. These sections are accessible from the main navigation bar and offer relevant information for both first-time and returning users.

The About page offers a detailed overview of the application, its purpose, and its key functionalities. It includes a structured table of contents covering topics such as data management, user and group administration, statistics, rankings, RSS, favorites, and group interaction. The Help page focuses on user support and contact options. It includes instructions for reaching out in case of technical issues via a support email, as well as a visual presentation of the team members responsible for the project. A link to full technical documentation is also provided.

2.7 Assumptions and Dependencies

This application assumes that users access the platform from a device with a stable internet connection and a modern web browser that supports HTML5, CSS3, and JavaScript. It is also assumed that JavaScript is enabled in the browser, as core functionality relies on client-side logic.

DrO website depends on the proper functioning of a backend service that exposes RESTful API endpoints. These endpoints are required for user authentication, data retrieval, drink list management, and statistics generation. If the backend service is unavailable or misconfigured, the application’s dynamic features may be partially or completely inaccessible.

Additional assumptions include server-side support for clean URLs (Pretty URLs) and proper CORS configuration to allow secure cross-origin requests between frontend and backend components.

3. External Interface Requirements

3.1 User Interfaces

The website uses a consistent layout across all pages, with a shared header that ensure a smooth and cohesive user experience. The header contains navigation links to the main sections of the platform — such as Home, Lists, Groups, and Statistics — and also includes icons for accessing the About and Help pages, along with the user profile area.

The entire interface is built with clarity and accessibility in mind. A bold, modern font is used throughout the platform to enhance readability and maintain a professional yet friendly appearance. Colors are carefully chosen to align with the theme of soft drinks, using vibrant tones on a clean background to keep the user engaged.

All pages are responsive and work seamlessly across devices. Main sections for authenticated users include persistent redirect buttons for easy navigation between modules, while secondary pages are accessible through direct navigation or through links embedded in the main interface. The layout is intuitive, and key actions such as adding to favorites, creating lists, or editing profile details are always clearly visible and accessible.

3.2 Hardware Interfaces

The application runs smoothly on a wide variety of devices, including desktops, laptops, tablets, and smartphones. DrinkApp is fully responsive, ensuring that the interface adapts automatically to different screen sizes and device types.

Since the application is entirely web-based, an active internet connection is required for it to function properly. No installation or specific hardware configurations are needed beyond a standard input device (keyboard, mouse, or touchscreen) and access to a supported web browser.

3.3 Software Interface

The frontend of the application is built entirely with standard web technologies—namely HTML, CSS and JavaScript—without relying on any third-party frameworks or libraries. The layout is modular, each page having its own folder structure following the "Pretty URLs" convention, and the code is split into semantic files for structure, style, and behavior.

Communication with the backend is achieved through a set of custom RESTful API endpoints that expose all major entities of the system (users, drinks, lists, favorites, groups, etc.). The frontend initiates HTTP requests using JavaScript’s fetch() API, and all requests and responses are exchanged in JSON format.

All essential operations—such as user registration, login, password recovery, data retrieval, list creation, profile updates, and favorite management—are performed through HTTP methods: GET (data retrieval), POST (creation), PATCH (partial update), and DELETE (removal). Each operation is scoped to a specific endpoint, designed to follow a consistent, predictable route naming convention.

The frontend securely handles authentication by storing a JWT (JSON Web Token) in memory and attaching it as a bearer token in the Authorization header of each request that targets a protected resource. This allows the backend to validate the identity and permissions of the user for each operation.

Error handling is also implemented on the client side, where the frontend gracefully displays feedback for failed requests such as invalid input, expired tokens, or missing data. The application is designed to offer fallback behaviors and user-friendly error messages wherever applicable.

3.4 Communication Interface

The DrO platform provides multiple communication mechanisms to facilitate user interaction with the system and the development team. First, a dedicated Help page is available, offering users direct support instructions, a support email (support@drosite.example.com), and a link to the complete technical documentation. This ensures users can easily report bugs, request assistance, or seek clarification.

In addition, the system includes a mailing mechanism integrated with the authentication flow: during account creation and password recovery, users receive confirmation or reset instructions via email. All system-generated emails follow a standard format and are sent through the backend using secure SMTP configuration.

While there is currently no direct contact form, communication is structured through predefined support channels, allowing users to access reliable help while maintaining platform security and simplicity.

3.5 API Interfaces

The application interacts with a custom REST API that handles all core functionalities related to user accounts, drinks, favorites, lists, groups, and rankings. The API follows standard HTTP methods such as GET, POST, PATCH, and DELETE, and communicates using JSON format.

All authenticated requests require a valid JSON Web Token (JWT) passed via the Authorization header. For example, endpoints such as /drinks/feed, /lists/:id, or /profile/update are protected and personalized based on the user's identity.

The API is modular and versioned, making it easy to extend in the future. A secure .env file is used on the frontend to store the base URL of the API, ensuring a separation between logic and configuration. This architecture enables the frontend to remain fully decoupled from the backend, simplifying development and deployment.

4. Structure

4.1 Entry Page

The Entry Page serves as the landing point for all users, whether authenticated or not. It presents a clean and minimal layout with two primary options: navigating to the Login or Signup pages. This page is implemented using standard HTML and CSS, with no external frameworks. Its role is to route users toward authentication flows in a clear and user-friendly manner.

4.2 Log-In Page

The Login Page allows existing users to authenticate using their registered credentials. It features input fields for email and password, both validated on the client side for correct format and completeness. Upon submission, the data is sent via a POST request to the authentication API endpoint, and if successful, a JWT token is returned and stored for future access. Error messages are clearly displayed beneath the respective fields in case of invalid input or failed authentication attempts. The page also includes a link to the password recovery flow for users who have forgotten their credentials.

4.3 Register Page

The Register Page enables new users to create an account by filling out a form that includes a username, email, and password. Each field is validated both in real-time and upon submission. The username must be 3–20 characters long and contain only letters, digits, or underscores, while the email is validated using a regular expression to ensure a proper format. The password must meet minimum security criteria. If all validations pass, the data is sent via a POST request to the backend, which creates the user account and returns a success response. Errors such as duplicate usernames or invalid data are shown inline, providing clear feedback to the user.

4.4 Home Page

The Home Page serves as the central hub for authenticated users, offering a personalized feed of drinks. The page retrieves drink data from the backend via asynchronous API calls and displays it in a grid-based layout, with each drink card showing key information such as the product name, nutrition grade, image, and available tags. Users can filter the feed using a sidebar with dynamic tag filters, which are also fetched from the backend. Sorting options such as “Order by name” or “Order by cathegory” are provided to improve navigation.

One of the core features of the Home Page is the drink discovery feed. Users can browse a curated list of drinks, which can be filtered based on predefined categories such as "sodas", "waters", "lemonade", or "teas". Additionally, drinks can be sorted by nutrition grade ("A" to "E"), or filtered by quantity (between 0 and 5000 ml). Sorting by other criteria such as "name", "brand", or "price" is also supported. Each drink panel includes detailed information that helps users decide whether to add the item to their favorites or explore it further. All these interactions are responsive and handled efficiently through JavaScript, using JWT authentication for secure access.

4.5 Lists Page

The Lists Page allows users to create and manage personalized collections of drinks. Each user can have multiple custom lists where they can add or remove drinks according to their preferences. Lists can be given a name and a visibility status (public or private), allowing users to share their recommendations or keep them private.

From the interface, users can perform actions such as renaming a list, deleting it, or modifying its contents by removing or adding drinks. Each list displays a panel of the included drinks, showing basic information (name, image, nutrition grade) to facilitate quick browsing. The backend provides all list-related data through protected API endpoints, and all modifications are saved in real-time via authenticated HTTP requests.

4.6 Group Page

The Group Page provides a collaborative space where users can join thematic groups focused on specific interests, such as healthy drinks, energy beverages, or sustainable brands. Each group has a dedicated section where users can view shared drink recommendations, browse other members’ lists, and interact through posts or comments.

Within a group, users can see the group's name, description, and drink collections associated with the community. Administrative users have access to special actions such as deleting inapropriate posts, removing members, or moderating content. All interactions are secured through authenticated API requests, and users must first join a group to gain access to its full features.

4.7 Join Group Page

The Join Group Page allows users to browse a list of public groups available on the platform. Each group entry displays key information such as the name or a short description. Users can search for groups using keywords or filter them by category to find communities that match their interests.

To join a group, users simply click the “Join” button, which triggers a secure request to the server and grants access to the corresponding Group Page. If a group is private, the request is sent for approval by an administrator. Once joined, users can actively participate in the group’s discussions, view shared drinks, and contribute to the collective knowledge of the group.

4.8 Profile Page

The Profile Page allows users to view and manage their personal account information. Key elements displayed on this page include the username, profile picture, email address, and a short personal description. This page serves as the central point for accessing user-related actions and customization.

Users can edit their profile details directly from this page, including updating their profile image, changing their email or username, modifying their description, and resetting their password. Additionally, the page provides two critical account options: the ability to securely log out and the option to permanently delete the account. All changes and actions are validated and executed through secure backend communication.

4.9 Profile-By-Id Page

The Profile by ID Page allows users to view the public profile of another registered user by accessing their profile in the groups page. The page displays key information such as the user’s profile picture, username, email address, and personal description. This view is read-only and tailored for exploration purposes.

In addition to basic user information, the page lists all public drink lists created by the user. Visitors can browse through these lists and explore their contents. A special feature available on this page is the “Copy List” button, which enables users to duplicate a selected list into their own account with a single click, encouraging sharing and discovery within the community.

4.10 Ranking Page

The Ranking Page displays the top 10 drinks that have been added to users’ favorite lists the most times. This leaderboard highlights the most appreciated beverages across the platform, based on cumulative user preferences.

Each ranked drink is shown with its image, name, and associated tags, offering visitors a quick overview of what the community values most. The ranking is updated dynamically to reflect real-time changes as users modify their favorite lists, allowing for a transparent and evolving popularity chart.

4.11 Stats Page

The Stats Page provides a visual representation of selected drinks through dynamic charts and summaries. Users can select specific beverages from the available dataset to generate tailored statistics based on their attributes.

The statistics include diagrams for quantity distribution, nutrition grade analysis, and tag frequency. These visual insights help users better understand the nutritional and categorical properties of drinks, offering a data-driven perspective on their selections.

4.12 About Page

The About Page offers a clear and structured overview of the application's purpose, features, and usage instructions. It is designed to help new users understand the scope and goals of the platform, as well as to navigate its main functionalities.

The content presented includes a structured table of contents, followed by a detailed project description. Sections cover aspects such as data management, user and group administration, statistics and export features, rankings, favorites, and group interactions. This makes the page both informative and accessible, contributing to a smooth onboarding experience.

4.13 Help Page

The Help Page is designed to offer users quick access to support resources and assistance. It includes a clearly visible section for Support & Contact, where users can find an email address for reporting bugs or asking technical questions. Additionally, a link to the full documentation is provided, guiding users to more detailed information.

Below the contact section, the page features a friendly presentation of the development team. Each member is represented with a playful drink-themed image, along with their name, reinforcing the visual identity and fun tone of the application. This helps humanize the platform and encourages users to reach out confidently for help.

5. System Features

5.1 Account Management

We have implemented a system that allows the user to sign up and log in. The user can also log out to end their session at any time. Authentication is managed through sessions and cookies. Once logged in, users can view and edit their profile details, including their password. If a logged-in user tries to access the login or signup pages again, they are redirected to the home page. Similarly, if an unauthenticated user attempts to access restricted pages, they are redirected to the login page. The admin page is also protected, and any unauthorized access attempts result in redirection to the home page. Passwords are stored in encrypted form to ensure the security of user data. Accounts can be edited or deleted by administrators from the admin page if any rules are violated.

5.2 Multi-Criteria Search System

The application provides users with multiple options to search and filter drinks based on various criteria. Users can filter drinks by predefined categories such as "sodas", "waters", "lemonades", and "teas". In addition, drinks can be sorted by nutrition grade (A, B, C, D, or E), by brand, name, price, or quantity. Users can also define a custom quantity range (from 0 to 5000 ml) to narrow down the results. All these filtering and sorting options are applied instantly to the drink feed, offering a flexible and user-friendly way to discover new beverages tailored to their preferences.

5.3 Statistics Export System

The statistics page provides an interactive and informative way for users to explore various data related to their selected drinks. They can analyze the distribution of drinks by quantity, nutrition grade, or tags through intuitive pie and bar charts, all built without external frameworks. These visualizations help users gain a deeper understanding of the characteristics of the drinks added in the application.

The user has the option to download the data as CSV or JSON files, directly from the graphical interface. This enables further processing of the data or integration into other external systems. For each type of displayed statistic, there is a dedicated download button, making it easy to save individual sets of relevant information.

Additionally, for the ranking page, which displays the most popular drinks based on how many users added them to their favorites, we have implemented an RSS feed. This feed provides an automated way to track updates in the top rankings, allowing applications or users to be notified whenever changes occur. Thus, our statistics and export system offers both accessibility and automation.

5.4 Leaderboards

In our application, we have implemented a dynamic leaderboard system that highlights the most appreciated drinks among users, based on how often they have been added to personal favorite lists. This system relies on a many-to-many relationship between users and drinks, allowing us to calculate and rank the most favorited items in real time. The leaderboard displays the top 10 drinks with the highest number of unique users who added them to their favorites, offering a clear view of the community’s preferences.

The purpose of this page is to help users discover popular drinks curated through collective taste, rather than editorial suggestion. The drinks are ordered descending by favorite count and updated automatically as user preferences evolve. This encourages interaction and exploration, offering users a way to find trending items and see which drinks are consistently appreciated across the platform.

5.5 User Friendly-Interface

Our platform was designed with a strong focus on user experience, providing an intuitive and accessible interface that ensures ease of navigation and usability for users of all ages and technical backgrounds. The layout is clean and logically structured, allowing users to quickly understand how to interact with different features, from browsing and filtering drinks to managing their personal lists and preferences. Interactive elements such as tooltips, hover effects, and clear feedback messages contribute to a seamless interaction with the platform.

Additionally, the website is fully responsive and adapts gracefully to various screen sizes and devices, including desktops, tablets, and smartphones. This guarantees a consistent and enjoyable experience across all platforms. All design decisions were made with accessibility and clarity in mind, resulting in an application that feels modern, lightweight, and pleasant to use, even during extended browsing sessions.

5.6 Administration Page

The administrator page integrates all functionalities available to a regular user, such as viewing and editing their own profile information – including username, email, description, and password – as well as the ability to delete their account or log out from the application. Beyond these common features, the administrator is granted elevated privileges that allow for full moderation and oversight of platform content.

Through this panel, administrators can view and permanently delete any user-generated content, including drinks and groups created on the platform. They also have the authority to manage user accounts by removing those who violate platform rules or engage in inappropriate behavior. These administrative capabilities are essential for maintaining the quality and integrity of the application and ensure a safe, respectful, and organized environment for all users. The interface for the admin section preserves the same intuitive and responsive design, making moderation tasks efficient and accessible.

5.7 REST API

Our application is built upon REST API principles to ensure modularity, scalability, and a clear separation of responsibilities between the frontend and backend. We implemented well-structured endpoints that follow a consistent and logical URL schema, allowing both regular users and administrators to interact with the system in a predictable and efficient manner.

For regular users, endpoints are available to handle authentication, account management, drink and list creation, viewing, editing, and deletion. Administrators have access to extended routes that provide functionality for managing global resources, such as deleting drinks, user accounts, and entire groups directly from the database. The API supports all standard HTTP methods—GET for fetching resources, POST for creating, PUT/PATCH for updating, and DELETE for removing entities.

5.8 MVC Arhitecture

The backend is built using the Model-View-Controller (MVC) architectural pattern, which ensures a clear separation of concerns between the data layer (models), the application logic (controllers), and the external interface (views, i.e., API responses). To further organize the codebase, we introduced service layers that handle business logic separately, making the system easier to maintain, extend, and test.

This structured design improves both scalability and clarity, allowing developers to isolate and modify specific functionalities without interfering with others. Combined with our RESTful API, the MVC architecture ensures better reliability, easier debugging, and readiness for future integrations, such as mobile clients or external services.

5.9 HTML and CSS validation

To ensure a high standard of quality and accessibility, we carefully validated all the HTML and CSS code using the official W3C validation tools. This process helped us identify and correct structural or syntactic issues, making the website more reliable and compliant with web standards.

In addition, we verified that the layout remains fully responsive across a wide range of screen sizes and devices, offering a consistent and pleasant user experience. We also focused on maintaining the security of the website and the protection of user data through good development practices, contributing to both usability and trust.

6. Other Nonfunctional Requirements

6.1 Performance Requirements

The application is optimized to ensure a smooth and responsive user experience, with minimal loading times across all supported devices. Static assets are efficiently served, and JavaScript execution is streamlined to reduce delays during navigation and interaction.

API requests are expected to return responses within 300ms in typical scenarios, with pagination and filtering mechanisms in place to prevent excessive data load. Backend endpoints return only the necessary data, which helps improve both speed and scalability as the number of users or items grows.

6.2 Security Requirements

The system implements multiple safety measures to ensure the protection of user data and prevent any form of unauthorized access or data leakage. Passwords are stored in encrypted form using secure hashing algorithms, and no sensitive user information is exposed in plain text, either on the client or server side. All form inputs are validated to prevent injection attacks or accidental data corruption.

In order to prevent unauthorized data access, the backend enforces strict access control. Each user only has permission to view or modify their own data. Administrative functions such as user deletion, group removal, or drink moderation are restricted exclusively to users with elevated privileges. These admin operations are separated from the regular user interface and require authentication with a valid JWT token tied to an admin account.

During the registration process, security-focused validation rules are applied to prevent abuse and ensure clean data. Usernames must contain between 3 and 20 characters and may only include letters, digits, and underscores. Email addresses are validated using regular expressions to ensure they follow the correct format. These constraints help prevent account spamming and contribute to maintaining the integrity of the user database.

6.3 Safety Requirements

Although the application does not deal with physical safety concerns, several measures are taken to ensure safe digital interactions. The platform limits the potential for misuse or instability by validating user input, enforcing strict data formats, and maintaining predictable system behavior. All critical operations are gated by appropriate checks to avoid unintentional data manipulation or system disruption.

User input is thoroughly validated both on the client and server side to prevent malformed data and accidental misuse. Error handling is implemented gracefully, so even in unexpected scenarios, the system provides informative feedback without compromising stability. Additionally, the frontend does not expose raw error traces or server-side messages, reducing the risk of information leakage.

To prevent database leaks or integrity issues, the system avoids exposing direct database identifiers and filters all user-submitted input. Although the platform does not involve physical safety, digital safety is a priority. Regular backend-side validations, limited input lengths, and careful handling of exceptions reduce the risk of service abuse or accidental system crashes.

6.4 Software Quality Attributes

The application prioritizes usability and responsiveness across all supported platforms. Its intuitive layout, clear labeling, and consistent design patterns ensure that users can easily navigate and interact with the system, regardless of technical background. All features are structured to provide quick feedback and minimal friction.

Maintainability is a key focus of the implementation. The frontend is developed using modular HTML, CSS, and JavaScript code without any external frameworks, which facilitates debugging and future extensions. The backend is structured into well-defined endpoints that follow RESTful conventions, making it easier to understand, test, and refactor components independently.

The system is also built with reliability and fault tolerance in mind. Failures in API communication, user input errors, or edge cases are handled gracefully, with clear user messages and fallback mechanisms where needed. As a result, the application remains stable and predictable even under atypical usage conditions. Additionally, all database queries are parameterized to prevent SQL injection attacks.

6.5 Business Rules

The application enforces a set of business rules to ensure data consistency, user experience integrity, and alignment with platform goals. Each user must create an account to access personalized functionalities such as creating drink lists, adding favorites, or posting new drinks. Usernames must be unique and conform to specific validation rules, while emails are checked for correctness using regular expressions.

Administrative actions, such as managing user accounts or moderating drinks and groups, are restricted to users with elevated privileges. These operations are separated from the general interface and require JWT-based authentication tied to an admin account. Additionally, duplicate list names are not allowed per user, and a drink cannot be added multiple times to the same list.

Tags and nutrition grades associated with drinks follow a standardized structure to facilitate filtering and statistics generation. Any action that modifies persistent data, such as deleting a user or removing a drink, is subject to validation to prevent data inconsistency or accidental loss.
